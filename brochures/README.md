# Inveenta Product Brochures — How to Build One

A playbook for producing a **print-ready A4 PDF brochure** for an Inveenta
product or solution. The first one built with this method is **Orchestrate**
(`orchestrate.html` / `orchestrate.pdf`, generated by `_build_orchestrate.py`) —
use it as the reference implementation.

The goal every time: a **single, self-contained HTML file** that renders to an
**exact-A4, multi-page PDF**, fully on **Inveenta CI**, with **original copy and
original mockups** (never a restyled copy of a client's draft).

---

## 1. What "good" looks like

- **Portable.** One `.html` file with the font embedded (base64) and every logo,
  diagram and UI mockup as inline SVG/CSS. No external images, no internet needed.
  It must survive being emailed and opened anywhere.
- **Exactly A4.** Each page is `210mm × 297mm`. The PDF must report `210 x 297 mm`.
- **On brand.** Inveenta CI (see §3) — not the client's palette, even if their
  brief used one.
- **Original.** You write the headlines, body and sample data yourself. Do **not**
  transcribe a client-supplied draft PDF — that gets rejected. Reuse the
  *positioning/scenario*, never the *wording*.

---

## 2. The toolchain

| Step | Tool | Notes |
|------|------|-------|
| Generate HTML | Python 3 (`_build_*.py`) | Emits one self-contained `.html`. |
| Embed font | `base64` of `public/fonts/montserrat-latin.woff2` | Inlined as a `data:` URL in `@font-face`. |
| Logo | `public/brand/mark-light.svg` (dark bg) / `mark-dark.svg` (light bg) | Read the file, inline the `<svg>`, size via CSS. |
| HTML → PDF | headless Chrome | `--headless --no-pdf-header-footer --print-to-pdf` |
| Verify fit | PyMuPDF (`fitz`) | Render the **PDF** to PNG and inspect — see §6. |

**Why a Python generator and not hand-written HTML?** It keeps copy/data in clean
Python structures, embeds the font and logos once, and lets you regenerate after
any tweak. Repeated content (feature cards, KPIs, kanban cards) is built from
lists, so edits are one-liners.

## 3. Inveenta CI rules (non-negotiable)

Tokens (from the website `src/index.css`):

```
--navy:        #16243B   structure, text, dark surfaces
--green:       #B7DA4C   THE single accent (lime)
--green-screen:#DBFC52   electric green — large numerals / accent words, ON DARK ONLY
--green-deep:  #A4C93C   graphic fills (progress bars), hovers
--green-wash:  #F1F6E1   faint tint (icon tiles)
--grey:        #5E5F64   secondary text, captions
--mist:        #E7EAEF   soft text on dark
--gradient-dark: linear-gradient(165deg,#424B5B,#2D3444,#1C2438)   hero/CTA/dark cards
```

- **Font:** Montserrat (embedded). Headings 700, tight tracking.
- **Logo:** the official `inveenta` wordmark (green *i*-tittle). `mark-light` on
  dark, `mark-dark` on light. Never redraw it.
- **🚫 The one rule people forget: NEVER green text on white.** Green on a light
  page may only appear as a *graphic* — the eyebrow dash, a heading underbar,
  progress-bar fills, dots, the agent send button. Coloured headline/accent words
  in green are allowed **only on dark backgrounds** (use `--green-screen`).
- **Brand patterns to reuse:**
  - *Eyebrow* = uppercase grey label (mist on dark) preceded by a 22×2px green dash.
  - *KPI / stat* = big `--green-screen` figure on dark; on white the figure is navy.
  - *Heading spark* on light pages = a short green underbar under the title
    (`h3::after`), not coloured text.
  - *Motif* = three dots in fixed order green · navy · white.
  - Dark sections (hero, CTA, comparison "with" card, AI band) use `--gradient-dark`.

Status colours (red `#CF5340` / amber `#D8932F`) are allowed **only** for genuine
risk/at-risk semantics in mockups — keep them sparse.

## 4. Page anatomy (the proven 5-page arc)

This structure works for any "platform/app" product. Swap the copy, keep the rhythm.

1. **Cover** — full-bleed `--gradient-dark`, faint grid texture, lime constellation,
   wordmark + tagline lockup, eyebrow, big headline (one clause in `--green-screen`),
   product name, lead, footer (`inveenta.com` · `Prepared for <name>`).
2. **The Challenge → The Outcome** — problem headline + intro; a two-card
   comparison (light "today" card with hollow bullets vs dark "with us" card with
   green bullets); three outcome columns with green underbars; a hub-and-spoke
   diagram of the core idea.
3. **What you get** — six feature cards (green-wash icon tile + navy stroke icon);
   then a dark "AI agent" band with tags and example prompts.
4. **A look inside** — three UI mockups built in HTML/CSS (dashboard, a board/kanban
   workspace, an agent chat). **This is the densest page — see §5/§6.**
5. **How we build it + CTA** — four delivery steps (numbered navy badges, green
   underbars), an "Engineered on" strip, then a dark CTA band with contacts.

Page furniture: each light page has `.pad` content + a pinned `.foot`
(`margin-top:auto`) showing `Inveenta · <Product>` and the page number.

## 5. Print layout: the rules that actually matter

- `@page { size: A4; margin: 0; }` and each `.page { width:210mm; height:297mm;
  overflow:hidden; }`. The fixed height + `overflow:hidden` is what makes one
  section = one sheet — **but it silently clips anything taller than 297mm.**
- `-webkit-print-color-adjust: exact; print-color-adjust: exact;` so dark
  backgrounds actually print.
- `@media print { .page { margin:0; box-shadow:none; page-break-after:always; } }`
  (on-screen the pages get a grey margin + shadow for preview; print strips them).
- **Cover full-bleed gotcha:** don't give the cover `height:100%` — a `%` height
  resolves against `<body>` (auto) and collapses to content height, leaving white
  at the bottom. Let it inherit the `.page` `height:297mm`.
- **Dense pages need per-page compression.** Page 4 (three mockups) and the lower
  half of pages 2–3 are tight. Scope overrides with a page class (e.g. `.p4 .lc{…}`)
  and tighten paddings/margins/font-sizes in millimetres until it fits (§6).
- **Flex gotcha that bit us:** a rule like `.agent__input span{flex:1}` also matches
  a sibling icon `<span>`, stretching it into an ellipse. Target the text with
  `> span:first-child` and pin the button with `flex:0 0 auto`.

## 6. Verifying fit — do NOT trust the browser screenshot

The single biggest time-sink: **Chrome's on-screen screenshot renders ≈ 0.8× and
lies about whether content fits A4.** Only the **printed PDF** is ground truth.
Always verify against the PDF with PyMuPDF.

### Render the real PDF pages to images

```python
import fitz
doc = fitz.open("orchestrate.pdf")
for i, p in enumerate(doc):
    p.get_pixmap(dpi=120).save(f"v{i+1}.png")   # then open & eyeball each page
```

### Strict bottom-edge clip check (catches the subtle clips)

A page **fits** only if its bottom ~13mm strip is mostly white — i.e. the footer's
whitespace zone is present, not a dark band / grey box / card jammed to the edge.

```python
import fitz
from PIL import Image
doc = fitz.open("orchestrate.pdf")
for i, p in enumerate(doc):
    p.get_pixmap(dpi=130).save(f"c{i+1}.png")
for i in range(doc.page_count):
    im = Image.open(f"c{i+1}.png").convert("RGB"); W, H = im.size
    strip = int(13/297*H); white = tot = 0
    for y in range(H-strip, H):
        for x in range(0, W, 4):
            r, g, b = im.getpixel((x, y)); tot += 1
            if min(r, g, b) > 250: white += 1
    frac = white/tot
    print(f"page{i+1}: {frac:.2f}  {'OK' if frac > 0.85 else '*** CLIPPING ***'}")
```

**Reading the result:** light pages with a footer should score **> 0.85**.
Pages that are *intentionally* dark/full-bleed at the bottom — the **cover** and a
page ending in a **dark CTA band** — will score low and that's **fine, not a clip**.
So: trust the score for normal light pages; eyeball the cover/CTA pages by hand.

### When a page clips, compress in this order

1. Shorten the longest body paragraph by a line (biggest, cheapest win).
2. Reduce inter-section margins (`margin-top` between blocks).
3. Reduce card/box padding and list `gap`.
4. Shrink any oversized diagram (`max-width` on the SVG) — but note a diagram's
   height is `max(text, svg)`, so shrink the **taller** side.
5. Drop font sizes 0.5–1pt as a last resort.

Re-render and re-check after each pass. Leave ~10mm of white above the footer.

## 7. Make a new brochure (checklist)

1. **Copy the generator:** `cp _build_orchestrate.py _build_<product>.py`.
2. **Rewrite the copy data** at the top (features, comparison bullets, outcomes,
   steps, prompts) and the **mockup data** (KPIs, departments, board cards, chat).
   All original — your words, your realistic sample numbers.
3. **Set** the output filename, the product name, the cover headline (pick the
   clause to highlight in `--green-screen`), and the contact block.
4. **Keep** the CI tokens, the logo lockups, the eyebrow/KPI/motif patterns and the
   page arc. Only content changes.
5. **Build → render → verify** (§2, §6). Iterate compression on dense pages.
6. **Personalisation:** the cover/CTA can carry `Prepared for <name>` and a tailored
   CTA line — that's recipient personalisation, not copied content.
7. **Clean up** scratch PNGs; commit `_build_<product>.py`, `<product>.html`,
   `<product>.pdf`.

### Definition of done

- [ ] PDF reports `210 × 297 mm`, correct page count.
- [ ] Bottom-edge check passes on every light page; cover/CTA eyeballed clean.
- [ ] Real `inveenta` wordmark present; no green text on white.
- [ ] All copy and mockup data original (not transcribed from any draft).
- [ ] Font embedded, diagrams inline — file opens offline with nothing missing.
